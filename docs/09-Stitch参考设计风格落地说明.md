## 1. 文档目标

本文件用于明确说明：  
当前项目的前台页面视觉风格，不是让开发人员自由设计，而是统一参考 `docs/stitch_` 目录下的页面稿进行落地。

本文件解决的问题包括：

- “参考 stitch 风格”过于抽象，不知道具体参考什么
- 不同模块开发人员对“工业风”“企业站风格”理解不一致
- 页面结构、卡片样式、标题形式、色彩气质容易被各自发挥
- 虽然已有 Token 规范，但缺少页面层的直接参照

## 2. 执行优先级

前台页面开发时，必须按以下优先级执行：

1. 先遵守任务书里的业务功能要求
2. 再遵守 [00-UI设计Token风格指南.md](E:\ZZY_PROJECT\crmeb_java\docs\00-UI设计Token风格指南.md)
3. 最后以 `docs/stitch_` 中对应页面的视觉风格作为页面层参考

说明：

- Token 文件负责“颜色、字号、间距、组件状态”的统一
- `stitch_` 参考稿负责“页面构图、信息层级、氛围和视觉语言”的统一

## 4. stitch 风格的核心视觉特征

根据参考稿与 `DESIGN.md`，当前项目前台页面应统一继承以下特征。

## 4.1 品牌气质

整体风格不是电商促销风，也不是纯内容媒体风，而是：

- 工业制造风
- 企业级官网风
- 技术型商城风
- 结构清晰、可信、克制的 B2B/B2C 混合风格

关键词：

- 工业蓝
- 专业灰
- 点状橙色强调
- 清晰分区
- 信息密度适中
- 少装饰、多结构

## 4.2 色彩气质

前台页面应继承以下色彩逻辑：

- 主色：深工业蓝
- 背景：浅冷灰、浅蓝灰
- 内容卡片：白色或很浅的冷色面
- 重点转化按钮：可用橙色/暖色做强调，但必须克制

执行要求：

- 页面大面积不使用高饱和红色做主视觉
- 不做营销活动页式大面积渐变
- 不做花哨的装饰背景
- 强调“专业、理性、稳定”，不是“热闹、促销、冲击”

## 4.3 版式结构

参考稿的共同特点是：

- 页面分区明确
- 模块边界清楚
- 卡片和列表结构规整
- 标题、说明、参数、操作按钮层级清楚

执行要求：

- 前台页面以“区块式布局”为主
- 每个区块都要有清晰标题或功能语义
- 产品、案例、下载等列表都要统一卡片化或列表化结构
- 不允许同一模块内既像企业站，又像活动页，又像内容资讯页

## 4.4 标题样式

参考稿中，大多数区块标题使用：

- 左侧蓝色竖条
- 标题文字
- 右侧“更多”或查看入口

执行要求：

- 产品推荐
- 新闻列表
- 案例模块
- 下载模块

以上模块优先复用这一标题模式，不要每个模块自创标题样式。

## 4.5 卡片风格

参考稿中的卡片具有以下共同特征：

- 白底或浅色底
- 轻边框
- 低阴影或无重阴影
- 小圆角到中圆角
- 信息区分层清晰

执行要求：

- 商品卡片
- 案例卡片
- 下载项卡片
- 联系方式信息卡

都必须维持这一套卡片气质，不允许做成重阴影大圆角营销卡。

## 4.6 图片使用风格

参考稿偏向：

- 工厂、设备、线缆产品、工业场景
- 清晰、真实、专业
- 明亮、高质感

执行要求：

- 首页 Hero 图
- 关于我们工厂介绍图
- 案例项目图
- 商品详情图

必须优先使用真实、清晰、可识别工业主题的图片。  
不允许使用过暗、过虚、纯氛围、看不清主体的素材。

## 4.7 商品信息表达风格

产品详情页参考稿已经明确了表达模式：

- 价格区
- 规格标签区
- 发货/运费/销量信息
- 技术参数表
- 图文详情
- 底部固定操作栏

执行要求：

- 商城模块商品详情页优先按这个结构组织
- 参数信息以表格或结构化列表展示
- 规格与卖点以标签形式辅助说明
- 不要把商品详情页做成资讯文章页

## 5. 与现有 Token 规范如何配合

`stitch_` 是页面参考，不是随意复制 HTML。  
真正落地时必须与现有 Token 规范结合。

配合原则如下：

- 页面层参考 `stitch_`
- 颜色、字号、边框、圆角、按钮状态遵守 `00-UI设计Token风格指南.md`
- 如果 `stitch_` 某些颜色与当前项目 Token 存在差异，以 Token 规范为准
- 如果 `stitch_` 某些 spacing 或组件写法不适配现有代码结构，应保留视觉语言而不是机械照搬

## 6. 不允许机械照搬的内容

开发人员不应直接把 `code.html` 原样复制进项目。  
不允许机械照搬的原因：

- 当前项目使用的是现有前端技术栈，不是 `stitch_` 中的静态 Tailwind 原型
- 原型页的交互、数据结构、路由状态未必和真实项目一致
- 有些颜色、图片、文案只是展示占位

因此，正确做法是：

- 继承页面结构
- 继承视觉语言
- 继承信息层级
- 按项目真实组件与数据方式实现

## 8. 验收时如何判断是否符合 stitch 风格

前台页面验收时，除功能正确外，还要检查以下内容：

### 8.1 气质是否一致

- 是否仍然是工业制造、企业站、专业商城风格
- 是否出现明显营销页、活动页、社交内容页视觉偏移

### 8.2 配色是否一致

- 是否以工业蓝、灰白中性色为主
- 是否出现过多不受控的高饱和颜色

### 8.3 页面结构是否一致

- 是否保留清晰区块化结构
- 是否保留卡片、列表、参数区、标题区的秩序感

### 8.4 信息层级是否一致

- 标题、说明、辅助信息、操作按钮是否层级清晰
- 参数和规格是否结构化，不混乱

### 8.5 图片与图标是否一致

- 是否仍然是工业主题图片
- 图标是否简洁、功能导向，而不是装饰导向

## 9. 开发人员禁止项

以下行为禁止：

- 不看 `stitch_` 参考稿，自己重新设计前台页面
- 把前台页面做成活动页或促销页风格
- 用大面积渐变、重阴影、大圆角改写页面气质
- 用与工业场景无关的图像素材
- 直接复制 `stitch_` 原型 HTML 当成正式项目代码
- 页面风格与其他模块明显脱节

## 10. 最终执行要求

从本文件生效后，所有前台页面开发必须满足：

1. 功能遵守模块任务书
2. 设计 Token 遵守 UI Token 指南
3. 页面视觉直接参考 `docs/stitch_`
4. 页面验收必须检查是否符合 `stitch` 参考风格

本文件的目的，是把“参考 stitch 样式风格”从一句口头要求，变成开发人员可直接执行的页面设计基线。
