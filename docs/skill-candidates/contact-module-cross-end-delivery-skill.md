# 商城联系模块跨端开发 skill

## 适用场景
- 在 CRMEB/商城类项目中新增一个完整业务模块，且同时涉及前台页面、后台管理、数据库表、接口、配置项。
- 模块本质是“收集表单 + 后台处理流转”，例如联系我们、线索登记、资料申请、案例咨询。
- 需求已经给出字段、状态、来源页、后台动作等明确边界，重点是按现有项目结构落地。

## 不适用场景
- 只改前端样式或文案，不涉及后端和数据库。
- 只是在已有模块上追加一个很小的字段，不需要新增页面、菜单或接口。
- 需求本身包含新系统接入，例如 IM、CRM、工单平台，这类超出单模块开发范围。

## 前置条件
- 先拿到任务书，明确字段定义、状态枚举、来源枚举、页面入口、后台处理动作。
- 先确认哪些能力可以复用，例如站点配置、通用表单、列表页、按钮组件、系统菜单。
- 本地要有可用的 Node/npm 和项目要求的 JDK 版本；老项目要特别确认 Lombok 与 JDK 的兼容性。
- 确认数据库可执行建表 SQL，并提前检查 `.gitignore` 是否会忽略 SQL 文件。

## 推荐工具/MCP 顺序
1. 读任务书和现有同类模块文档。
2. 用 `rg` 搜索现有配置项、接口风格、管理页结构、前端入口位置。
3. 先补后端模型、请求对象、响应对象、service、dao、controller。
4. 再接前台页面、提交接口、来源页跳转入口。
5. 再接后台配置页、列表筛选、状态处理弹窗、菜单路由。
6. 补 SQL 和模块说明文档。
7. 最后跑构建、接口校验、来源埋点校验、提交流程校验。

## 标准流程
1. 先把任务书里的“字段、状态、来源、页面入口、后台动作”整理成固定清单，不在开发中途临时扩张需求。
2. 优先复用项目里已有配置项和通用能力，避免为单模块再造一套配置体系。
3. 后端先定义数据库表和 Java 模型，再补 request/response，最后补 controller/service/dao，保证字段命名先稳定。
4. 如果前台接口允许游客提交，及时把接口加入匿名访问白名单，否则页面联调阶段会被登录拦住。
5. 前台页面只做任务书要求的能力，入口页要显式传递 `sourcePage` 一类来源参数，避免来源丢失。
6. 后台页同时承担两类职责：联系配置维护、留言列表和状态流转；列表筛选条件和详情字段要与表结构一致。
7. 提交前对前台、后台、后端三处校验规则做一次对齐，尤其是手机号、备注长度、必填字段、状态值。
8. 补一份模块说明文档，记录配置键、表字段、页面流转、状态含义和验证方式。
9. 构建和验证时，至少覆盖一条完整链路：前台读取配置 -> 用户提交 -> 后台可见 -> 后台更新状态。

## 关键校验点
- 任务书、数据库、请求对象、响应对象、后台列表列名、前台表单字段名必须一致。
- 状态枚举和来源枚举必须固定，不能边开发边新增未约定值。
- 前台匿名接口必须在网关或 WebConfig 中放行，但后台接口不能放开。
- 前台、后台、后端的手机号校验和长度限制必须一致。
- 从商品详情、个人中心等入口跳转时，来源页参数必须落库，不能只在页面上带着不入库。
- 配置项键名必须统一，前台读取和后台保存要对上。
- SQL 文件如果被 `.gitignore` 忽略，要额外确认是否已手工导入，或是否需要 `git add -f`。
- 构建成功不能只看前端；Java 老项目要用项目要求的 JDK 再跑一次编译。

## 常见失败与处理
| 现象 | 原因 | 处理 |
|---|---|---|
| 前台页面能打开，但接口返回未登录或 403 | 游客接口没有加入匿名访问白名单 | 在前台服务的 WebConfig 中补充对应接口排除规则，再重新联调 |
| 留言提交成功，但后台来源页为空或全是默认值 | 入口页没有传 `sourcePage`，或后端没有持久化来源字段 | 在跳转入口显式传来源参数，并在请求对象、表字段、保存逻辑里全链路打通 |
| 后台能保存联系方式，但前台仍显示空数据 | 配置键名不一致，或前台/后台读取的不是同一组配置 | 把配置键收敛到常量，文档里明确每个键的用途 |
| 一端手机号能通过，另一端报校验失败 | 前台、后台、后端用了不同正则或不同长度限制 | 统一手机号规则，连同输入框 `maxlength` 一起对齐 |
| `git status` 看不到新建 SQL | 仓库 `.gitignore` 忽略了 `*.sql` | 明确改为手工导库，或在需要纳入版本库时使用 `git add -f` |
| 管理端构建通过，但后端编译报 Lombok/JDK 错误 | 老版本 Lombok 与当前 JDK 模块访问不兼容 | 切到项目要求的 JDK 版本重新编译，必要时再评估升级 Lombok |

## 安全边界
- 不在文档里记录真实账号、密钥、数据库连接串、手机号等敏感信息。
- 不擅自增加新状态、新来源、新业务流转，除非任务书明确要求。
- 不把“联系我们”做成在线聊天系统，避免把轻量表单模块做成新业务系统。
- 不回退与当前需求无关的脏改动，尤其是在多人并行开发分支上。
- 不把后台处理接口开放给匿名用户。
