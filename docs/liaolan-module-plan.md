# 辽缆项目模块开发规划

生成日期：2026-05-07

## 一、现状判断

### 1. 商城模块是不是已经有了

有，而且主链路已经比较完整，不是从零开发。

当前项目已具备的商城基础能力包括：

- 商品管理：`api/admin/store/product/*`
- 商品分类：`api/admin/category/*`
- 商品规格/属性：`StoreProductAddRequest.attr`、`attrValue`
- 购物车：`api/front/cart/*`
- 预下单/下单：`api/front/order/pre/order`、`/create`
- 支付：`api/front/pay/payment`，并已对接微信支付服务层
- 订单管理：前后台订单列表、详情、取消、退款、发货、物流查询
- 会员基础：手机号登录、用户地址、订单记录、收藏等
- 内容模块底座：文章、文章分类、附件管理、页面装修

结论：  
“产品商城模块”应以**二开改造**为主，不建议重做电商底座。

### 2. 产品支持多级分类吗

支持。

依据：

- 分类模型 `eb_category` 有 `pid` 和 `path` 字段，天然是树形结构
- 后台已提供 `api/admin/category/list/tree`
- 分类服务 `CategoryServiceImpl` 已按父子关系和路径处理树结构
- 商品请求 `StoreProductAddRequest.cateId` 为“分类 id 组合，逗号分隔”

结论：  
系统支持多级分类，但当前更偏通用零售商品分类。  
如果用于电线电缆，需要再补一层**业务分类体系设计**，例如：

- 一级：电力电缆 / 控制电缆 / 特种电缆
- 二级：阻燃 / 耐火 / 低烟无卤
- 三级：型号 / 电压等级 / 应用场景

## 二、总体实施建议

建议按“先商城闭环，再做内容与线索模块”的顺序推进：

1. 产品商城模块
2. 关于辽缆模块
3. 联系我们模块
4. 客户案例模块
5. 公司新闻模块
6. 资料下载模块

原因：

- 商城是交易主链路，优先级最高
- “关于辽缆 + 联系我们”能尽快补齐品牌和转化入口
- 案例、新闻、下载本质上都是内容运营与线索收集扩展，可复用文章/附件能力

## 三、各模块开发规划

## 模块 1：产品商城模块

### 目标

在现有 CRMEB 商城基础上，改造成适配电线电缆行业的展示、选型、下单、支付、履约闭环。

### 现有基础

- 商品、分类、购物车、订单、支付、会员、地址能力已具备
- 支持商品规格属性
- 支持订单状态流转、后台发货、物流单号录入
- 支持微信支付主链路

### 主要改造点

#### 1. 产品展示改造

- 重构商品详情字段结构，适配电线电缆产品
- 补充展示区块：
  - 产品简介
  - 型号说明
  - 技术参数
  - 执行标准
  - 适用场景
  - 资质证书
  - 下载资料
- 调整列表页筛选项：
  - 分类
  - 型号
  - 电压等级
  - 导体材质
  - 阻燃/耐火特性

#### 2. 分类体系规划

- 梳理行业分类树
- 明确商品“主分类 + 辅助筛选属性”的边界
- 避免把所有筛选条件都堆进分类树

推荐做法：

- 分类用于导航
- 属性用于筛选
- 规格用于 SKU 组合

#### 3. 规格与 SKU 改造

- 现有规格体系复用
- 补充适配电缆场景的规格维度：
  - 长度
  - 型号
  - 芯数
  - 截面积
  - 颜色/护套
- 明确哪些作为 SKU，哪些作为参数

#### 4. 下单与支付

- 保留 CRMEB 下单与微信支付主链路
- 核对微信支付商户配置、回调、证书、退款链路
- 明确价格逻辑：
  - 按卷售卖
  - 按米售卖
  - 阶梯价
  - 报价型商品是否允许 0 元下单或仅询盘

#### 5. 订单履约

- 保留现有订单状态流转
- 补充后台字段：
  - 发货说明
  - 物流公司
  - 物流单号
  - 线缆批次/出厂编号（如需要）

#### 6. 会员中心

- 复用现有订单、地址、登录基础
- 补充：
  - 个人资料完善
  - 发票信息
  - 企业客户字段（公司名、联系人、税号）

### 交付拆分

#### 第一阶段

- 分类体系设计
- 商品字段设计
- 前台商品列表/详情改版

#### 第二阶段

- 规格 SKU 适配
- 购物车、下单、订单页联调
- 微信支付联调

#### 第三阶段

- 后台订单履约增强
- 会员中心增强

## 模块 2：客户案例模块

### 目标

建设面向行业客户的案例展示模块，形成“案例内容 + 关联产品 + 询盘转化”链路。

### 现有基础

- 已有文章模块
- 已有文章分类
- 文章已支持 `productId` 关联商品

### 建议方案

优先采用“**文章模块二开为案例模块**”方案，不单独新建复杂内容引擎。

### 主要开发点

- 新增案例分类体系：
  - 行业
  - 区域
  - 项目类型
- 案例详情结构化：
  - 项目名称
  - 项目背景
  - 使用产品
  - 施工/供货说明
  - 实施效果
  - 图片集
- 前台支持筛选、列表、详情
- 案例详情页增加关联产品和联系方式入口

### 交付拆分

#### 第一阶段

- 案例内容模型确认
- 文章分类复用或新增案例分类

#### 第二阶段

- 前台案例列表、筛选、详情
- 后台案例发布与编辑

#### 第三阶段

- 关联产品跳转
- 询盘引导与线索沉淀

### 主要开发点

  - 企业动态

- 支持封面图、摘要、图文详情、分享优化
- 后台增加更贴近新闻运营的表单体验

### 交付拆分

#### 第一阶段

- 文章分类整理为新闻分类
- 前台新闻列表、详情页设计

#### 第二阶段

- 后台发布体验优化
- SEO 字段、分享字段梳理

#### 第三阶段

- 首页新闻推荐位

## 五、关键技术与业务注意事项

### 1. 商品分类不要等于筛选属性

分类树适合导航，属性更适合筛选。  
电线电缆商品如果把型号、电压等级、芯数都做成树分类，后期维护会很重。

### 2. 资料下载和联系我们都涉及线索沉淀

建议统一规划“线索表 / 咨询表 / 下载登记表”的关系，避免后面重复采集。

### 3. 微信支付现成但必须重新联调

代码底座已有，不代表商户配置、证书、回调、退款在你当前环境可直接用。  
上线前必须做商户参数、回调域名、证书与支付场景的整体验证。

### 4. 内容模块尽量复用文章与附件

案例、新闻、关于页、资料中心都应尽量复用现有内容底座，减少新表和新后台复杂度。

## 六、结论

这个项目不适合“全部重做”，适合“以现有商城为底座，按行业需求做业务化二开”。

明确判断如下：

- 产品商城：已有主链路，重点做行业适配
- 多级分类：已支持
- 客户案例：建议基于文章模块扩展
- 公司新闻：建议直接复用文章模块
- 资料下载：建议基于附件管理扩展为业务资料中心
- 关于辽缆：建议做单页专题化配置
- 联系我们：建议新增留言与线索管理能力

后续如果继续推进，建议下一步先输出：

1. 商品分类与属性模型设计
2. 六大模块的数据表清单
3. 前后台页面清单与接口清单
