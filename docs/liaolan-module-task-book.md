## 1. 文档目的

本任务书用于把辽缆项目按模块拆分给不同开发人员并行推进。目标不是重做 CRMEB，而是在现有 CRMEB Java 版基础上完成电线电缆行业化二开。

文档重点覆盖：

- 每个模块当前项目里已有的基础能力
- 每个模块建议的开发边界
- 每个模块的开发内容、开发方案、接口/数据影响
- 每个模块的验证方案、联调方案、交付物
- 各模块之间的依赖关系与排期建议

## 2. 项目现状判断

### 2.1 当前项目不是空白官网，而是成熟商城底座

项目已经具备完整电商主链路，核心证据如下：

- 商品后台：`admin/src/views/store/*`
- 商品后台 API：`admin/src/api/store.js`
- 订单后台：`admin/src/views/order/*`
- 订单后台 API：`admin/src/api/order.js`
- 前端商品与下单：`app/api/store.js`、`app/api/order.js`
- 前端商品页：`app/pages/goods/*`
- 文章内容体系：`admin/src/views/content/article/*`、`admin/src/api/article.js`
- 前端新闻页：`app/pages/news/*`
- 后端前台接口：`crmeb/crmeb-front/src/main/java/com/zbkj/front/controller`
- 后端后台接口：`crmeb/crmeb-admin/src/main/java/com/zbkj/admin/controller`

### 2.2 多级分类已支持

现有分类体系不是单级分类，已经具备树形能力：

- 后台分类 API：`/admin/category/list/tree`
- 前台分类 API：`/api/front/category`、`/api/front/categorybypid/{pid}`
- 后端分类控制器：`CategoryController`
- SQL 基础表：`eb_category`

这意味着“产品商城模块”的重点不是从零造分类，而是设计适合电线电缆的分类树和筛选维度。

### 2.3 产品详情当前是富文本 HTML

商品详情当前就是 `content` 富文本：

- 后台商品编辑页使用 `Tinymce`：`admin/src/views/store/creatStore/index.vue`
- 前端详情页通过 `jyf-parser` 渲染 HTML：`app/pages/goods/goods_details/index.vue`
- 后端请求对象存在 `content` 字段：`StoreProductAddRequest`

所以现状适合先承接“图文详情”，但不适合直接承接电线电缆行业常见的结构化参数展示，需要补结构化字段或扩展模型。

### 2.4 新闻模块已有现成底座

新闻本质上可直接复用文章模块：

- 后台文章管理：`admin/src/views/content/article/list.vue`、`edit.vue`
- 后台文章 API：`admin/src/api/article.js`
- 前端新闻页：`app/pages/news/news_list/index.vue`、`news_details/index.vue`
- 后端前台文章接口：`ArticleController`
- SQL 基础表：`eb_article`

### 2.5 资料中心、案例、关于页、联系我们都应以复用为主

现有系统已经具备可复用的内容与附件基础：

- 附件管理：`admin/src/api/system.js`、`admin/src/api/systemSetting.js`
- 上传接口：`/admin/upload/image`、`/admin/upload/file`
- 前台文件上传能力：`UserUploadController`
- 页面装修/单页底座：前台存在 `PageDiyController`

因此除“下载权限、线索收集、在线留言”外，大部分内容展示不建议重新建设一套 CMS。

## 3. 模块拆分原则

### 3.1 推荐人员分工

建议按 6 个业务模块拆负责人，每个负责人同时协调前后端与测试，不按“前端/后端”切块，以减少推诿。

建议角色模型：

- 模块负责人：1 人，对需求、数据模型、接口、验收负责
- 后端支持：1 人，可兼任多个模块
- 前端支持：1 人，负责 `admin` 和 `app`/H5 页面
- 测试支持：1 人，按批次验收

### 3.2 推荐模块顺序

1. 产品商城模块
2. 联系我们模块
3. 关于辽缆模块
4. 客户案例模块
5. 公司新闻模块
6. 资料下载模块

原因：

- 商城是主交易链路，优先级最高
- 联系我们和关于页直接支撑品牌展示与留资
- 案例、新闻、下载属于内容运营和询盘转化增强

## 4. 模块 1：产品商城模块任务书

### 4.1 模块目标

在 CRMEB 现有商品、购物车、订单、支付、会员链路上，完成适配电线电缆行业的商品展示、规格选型、下单支付、订单履约和会员中心增强。

### 4.2 当前基础

已存在：

- 商品管理：`/admin/store/product/*`
- 商品分类：`/admin/category/*`
- 商品规格/SKU：后台商品编辑页已支持多规格
- 购物车：`/api/front/cart/*`
- 预下单/下单：`/api/front/order/pre/order`、`/api/front/order/create`
- 支付：`/api/front/pay/payment`
- 后台订单流转：`/admin/store/order/*`
- 前端会员/地址/订单：`app/api/user.js`、`app/api/order.js`

结论：本模块是“行业化改造”，不是“从零开发商城”。

### 4.3 开发负责人边界

负责人只负责本模块内以下范围：

- 商品分类体系设计
- 商品字段与详情结构设计
- SKU 规格设计
- 商品列表/详情/搜索/筛选
- 购物车、下单、支付联调
- 订单履约字段增强
- 会员中心与地址相关增强

不负责：

- 分销逻辑改造
- 短信平台接入
- 内容模块文章体系通用改造

### 4.4 详细开发内容

#### 4.4.1 产品展示

开发内容：

- 设计电线电缆多级分类树
- 设计商品详情结构
- 增加行业属性展示区
- 增加列表筛选维度

建议字段拆分：

- 通用字段继续复用 `eb_store_product`
- 结构化行业字段建议新增扩展表，例如：
  - 产品型号
  - 电压等级
  - 导体材质
  - 芯数
  - 截面积
  - 护套材质
  - 阻燃等级
  - 执行标准
  - 适用场景
  - 认证证书

开发方案：

- 商品图文详情继续沿用现有 `content` 富文本
- 参数、标准、证书、下载资料不要继续塞进富文本，改单独结构化
- 后台商品编辑页在 `admin/src/views/store/creatStore/index.vue` 扩展表单分组
- 前台商品详情页在 `app/pages/goods/goods_details/index.vue` 新增参数区、证书区、资料区

#### 4.4.2 在线购买

开发内容：

- 明确哪些维度进入 SKU
- 明确哪些维度只是展示参数
- 支持选择规格、加入购物车、立即购买

SKU 设计建议：

- 进入 SKU：长度、型号、芯数、截面积等真正影响价格/库存的维度
- 保持参数：执行标准、适用场景、说明文案等不影响库存的维度

开发方案：

- 复用现有多规格能力
- 后台商品编辑页继续走现有规格组合逻辑
- 前台商品详情页继续走现有商品规格选择组件
- 必须先完成“行业 SKU 规则表”，再录入商品

#### 4.4.3 微信支付集成

开发内容：

- 核对微信支付商户配置
- 核对回调地址和签名
- 核对支付成功后订单状态变更

开发方案：

- 代码层先复用现有支付链路
- 环境层补全商户号、证书、回调域名
- 验证 H5/公众号/小程序分别能否拉起支付，按你实际使用端决定范围

#### 4.4.4 订单管理

开发内容：

- 支持待付款、已付款、已发货、已完成、已取消
- 后台录入物流公司、物流单号、发货备注
- 如业务需要，增加批次号/出厂编号

开发方案：

- 复用现有后台订单页：`admin/src/views/order/index.vue`
- 复用发货页：`admin/src/views/order/orderSend.vue`
- 若线缆业务需“批次追踪”，新增订单扩展字段或订单商品扩展字段

#### 4.4.5 会员中心

开发内容：

- 用户订单记录
- 收货地址管理
- 个人资料维护
- 企业客户补充资料

开发方案：

- 复用现有用户中心 API 与页面
- 增加企业字段：
  - 公司名称
  - 联系人
  - 联系电话
  - 税号
  - 开票信息

### 4.5 技术方案

后端：

- 商品主表尽量少动
- 行业字段优先新建扩展表，不污染通用商城主表
- 订单附加履约字段如果少，可优先扩展订单表；如果多，建议订单扩展表

后台：

- 商品录入页按“基础信息 / 行业参数 / 规格库存 / 图文详情 / 资料附件”分区
- 避免把电缆业务字段继续堆在富文本里

前台：

- 商品列表页强化筛选
- 商品详情页强化参数、规格、资料下载、询盘入口

### 4.6 验证方案

功能验证：

- 能创建三级分类商品
- 能录入图文详情和结构化参数
- 能选规格加入购物车
- 能生成订单
- 能支付成功
- 后台能发货并录入物流单号
- 前台能查看订单流转

数据验证：

- SKU 价格、库存与选项匹配
- 商品详情富文本与结构化参数都能正确展示
- 分类筛选和搜索不串数据

回归验证：

- 现有普通商品不受影响
- 购物车、订单、支付老链路不回归

### 4.7 交付物

- 分类方案文档
- 商品字段字典
- 商品后台页面改造
- 商品前台页面改造
- 下单支付联调记录
- 验收用测试商品数据

### 9.1 模块目标

形成标准咨询入口，支持在线留言、客服触达、一键导航和线索状态管理。

### 9.2 当前基础

已存在：

- 用户体系
- 前端表单能力
- 电话/客服按钮基础组件能力

缺失：

- 业务级留言表
- 后台留言处理流程
- 留言状态流转

### 9.3 开发负责人边界

负责：

- 留言表单
- 留言后台管理
- 客服入口整合
- 地址/地图导航
- 线索状态流转

### 9.4 详细开发内容

开发内容：

- 在线留言字段：
  - 姓名
  - 手机号
  - 公司名称
  - 咨询内容
  - 来源页面
- 后台处理状态：
  - 待处理
  - 已联系
  - 已转商机
  - 无效线索
- 客服入口：
  - 企业微信二维码
  - 微信客服链接
  - 电话拨打
- 一键导航：
  - 公司地址展示
  - 地图跳转

开发方案：

- 新建留言业务表，不复用短信记录或其他日志表
- 后台新增留言管理页
- 前台联系我们页与其他模块底部都可复用咨询入口
- 案例详情页、资料下载页、商品详情页可埋点来源页面

### 9.5 验证方案

- 前台可提交留言
- 留言必填校验正确
- 后台能看到留言记录
- 后台能修改处理状态
- 客服二维码/电话/导航入口可用
- 能区分线索来源模块

### 9.6 交付物

- 留言表设计
- 后台留言管理页
- 前台联系我们页
- 公共咨询组件
- 线索流转说明

## 10. 模块间依赖关系

### 10.1 强依赖

- 客户案例模块依赖产品商城模块中的“商品详情与商品可跳转能力”
- 资料下载模块与联系我们模块都依赖统一线索字段规划
- 关于辽缆模块和联系我们模块依赖统一公司基础信息来源

### 10.2 建议统一设计的公共能力

这些能力不要由某个模块私自定义，建议先统一：

- 公司基础资料字典
- 线索字段字典
- 下载登记字段字典
- 商品行业参数字典
- 分类命名规范
- 图片与文件命名规范

- 已知问题列表

## 12. 风险点

### 12.1 最大风险不是代码，而是业务模型不先定

尤其以下几项如果不先定，开发一定反复返工：

- 电线电缆分类树
- SKU 维度和参数维度边界
- 下载留资字段
- 留言线索状态
- 案例和新闻分类规则

### 12.2 不要把结构化业务数据继续塞进富文本

当前 CRMEB 富文本能力很强，但行业系统不能完全依赖富文本，否则会带来：

- 后台录入混乱
- 前台展示不统一
- 无法做筛选和检索
- 无法做统计和复用

### 12.3 资料下载与联系我们会形成重复线索

必须在方案层统一“线索表 / 下载登记表 / 留言表”的关系，否则后面客户数据会分散。

## 14. 结论

这个项目最合理的做法是：

- 商城主链路全面复用 CRMEB
- 内容体系尽量复用文章与附件
- 对电线电缆行业真正关键的部分做结构化扩展
- 每个模块按“负责人制”推进，不按技术栈切人

1. 商品行业数据模型设计文档
2. 线索与下载登记数据模型设计文档
3. 六大模块页面与接口清单文档
